# 装甲板追踪节点

订阅armor_detector或fake_armor_publisher(fake_ap)发布的装甲板信息和serial发布的自瞄模式、云台状态，发布云台控制命令。

使用同济大学开源的MPC规划器进行云台控制。

## 关于整车状态观测器

使用重投影误差约束的因子图优化实现，转速和位置的拟合非常稳。消除了PNP引入的误差。但由于因子图的数值敏感性，其对敌方车辆半径的拟合并不理想。也可能是我参数没调好。建议给法向噪声调大一点。

每一个装甲板位姿连接四个单点重投影约束因子（即四个角点），通过整车状态因子连接到约束过自由度的整车状态（即三维位置和偏航角），相邻两帧的位置和偏航角由平移因子和旋转因子相连，这两个因子同时连接到优化的速度和角速度变量，相邻两帧的速度由速度一致因子相连（近似匀速运动）。在增量优化过程中，由整车状态因子传递速度一致因子带来的先验，约束以PNP结果为初值的重投影误差优化过程，减少帧间识别抖动带来的噪声。

## 关于选板

假设敌方车辆半径25厘米，两转每秒的情况下，瞄准中心和最近装甲板两种情况的子弹飞行时间内敌方小陀螺转动角度差距不超过10度，故可先计算瞄准distance减去radius距离下的子弹飞行时间做预判（几十毫秒，敌方旋转几十至几百度），再进行装甲板选择（避免实际击打和瞄准错开）。

因为选择击打此时最面对的装甲板进行击打，锁定这块装甲板后进行迭代飞行时间计算（不在每次迭代时重新选板，避免迭代时反复横跳无法收敛）。如果出现“迭代后期锁定装甲板预判旋转至背对”的极端情况（基本不可能，除非观测器发散出现异常值或者对面真的能转到每秒十圈以上），直接放弃该帧瞄准（而不是重新锁定别的装甲板，因为这种状况已经违背物理常识了），至于斜对着抑制开火避免打舵的火控就交给切板跳变时MPC撞到加速度限制来完成了。

对于将近静止状态下斜45度附近的目标，为避免观测器噪声导致的帧间选板反复，对非锁定装甲板面对角度相对锁定装甲板面对角度差的绝对值设置一个死区（即当非锁定装甲板明显更加面对时才进行切板）。这会涉及到在帧间缓存选板，需要判断下什么时候重置这个缓存。

## 关于弹道解算

需要一个传入target_state，返回瞄准命令opt的接口。更新选板和预判时间的缓存。

在内部需要区分三段预判用时间：算法延迟（通过时间戳相减得到）、飞行时间延迟（迭代计算）、拨盘响应延迟（相信电控），但对外只需暴露一个用于获取当前瞄准的armor_id和总共的预判时间的接口就好了（在这个方法内使用智能锁保证线程安全，而不是直接public一个atomic，太傻逼了）。

火控类会在弹道解算之后调用，通从弹道类获得的选板和预判时间信息及该帧目标状态，还原现场，考虑装甲板投影大小和散布锥形计算yaw和pitch的火控阈值。同时，弹道类提供的这两个信息也会被用于绘制调试信息。
